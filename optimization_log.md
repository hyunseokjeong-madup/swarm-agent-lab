# 스웜 최적화 운영 일지 (Optimization Journal)

이 파일은 **스웜 최적화 기획자(Planner)**가 매 세대 남기는 기록이다.
각 세대마다: 집단(designs) → 스웜 평가(fitness) → 분석 → 다음 세대 설계 근거.

- 벤치마크: 10문제, 코드로 검증된 정답(`benchmark/answers.json`). 결정론적 채점.
- 적합도 측정: closed-book(도구/코드 실행 금지) 추론. 매뉴얼의 순수 추론 향상력 측정.
- 엘리트 보존: 매 세대 최고 적합도는 떨어지지 않는다.

## Generation 0 — 초기 집단 정의 (2026-06-27)

초기 집단 4종 (가설 공간을 넓게 커버):
| 설계 | 전략 가설 |
|------|-----------|
| C0_baseline | 매뉴얼 없음. 대조군. |
| C1_decompose | 명시적 분해 + 단계별 풀이. |
| C2_selfverify | 분해 + 독립적 방법으로 자기검증(필수). |
| C3_adversarial | 분해 + 자기 답을 적대적으로 반박(함정 플래깅, 재계산). |

가설: 검증·적대적 검증이 추가될수록 정확도가 오를 것이다. 단, 과한 절차는
짧은 문제에서 self-confusion을 유발할 수 있다(검증 필요).

다음: Generation 1 스웜 평가 실행 → 적합도 측정.

### Planner note: Gen1 스웜 발사 기록
- 방식: Workflow 스웜, 4 설계 × 10 문제 = **40개 평가 노드** 병렬.
- 각 노드: closed-book(도구/코드 금지) 추론 후 StructuredOutput로 답 1개 반환.
- 1차 시도(args 전달) 실패: `args` 미전달(undefined) → 데이터를 스크립트에 직접 내장(`swarm_eval.js`)하여 재발사. (기록: 스웜 입력은 args 의존 말고 self-contained가 안전.)
- 결과 수집 후 `score.py`로 결정론적 채점 예정.

### Designer note: Gen2 변이 연산자 툴킷 (사전 준비)
실패 문제군에 따라 적용할 매뉴얼 변이 규칙:
- 산술 실패(P1/P4/P7) → "큰 합/반복 계산은 자릿수별로 분할, 중간합을 5개 단위로 검산".
- 확률/분수(P2/P6/P9) → "경우의 수는 분자·분모를 따로 세고, 마지막에 GCD로 약분 확인".
- 함정(P6) → "조건부확률은 베이즈 표(4칸: TP/FP/FN/TN)를 그려라".
- 정밀독해(P5) → "단어 카운트는 토큰을 번호 매겨 나열 후 표시".
- 논리(P3) → "제약을 위치 격자에 채우고, 강제되는 칸부터 확정".
- 과적차 경계: 규칙이 짧은 문제에서 self-confusion 유발하면 제거.

### Gen1 결과 (closed-book, 10문항, 1시행) — lessons & missteps
- C0=10/10, C2=10/10, C3=10/10, **C1_decompose=8/10**.
- **misstep 발견**: 순수 분해가 무거운 산술(P1 소수합, P4 콜라츠합)에서 baseline보다 틀림.
- **lesson**: 검증(C2/C3)은 거의 항상 이득, naive 분해는 계산위생 동반 필수. → KNOWHOW.md B항에 기록.
- 10문항은 변별력 부족 → **20문항으로 확장**(P11~P20: 코프라임수/조건부확률/2^200자릿수합/두자녀함정/LIS/편집거리/동전교환/끝자리0). 모두 코드 검증.

## Generation 2 — 112-design 스웜 토너먼트 (2026-06-27)

### 집단 생성 (Designer 스웜, 14계열 × 8 = 112)
- Anthropic "Building Effective Human-Agent Teams" 블로그 노하우 흡수 → KNOWHOW.md A항.
  핵심: Doer-Verifier / 역할로스터 / 반성사이클 / 계층적에이전트 / 병렬워크스트림 / 암묵지외부화.
- 계열: 분해+계산위생, 독립검증, 적대검증, 함정체크리스트, 제약전파, 계산위생, 추정후검증,
  기지문제환원, 제일원리, 이중처리, 메타인지, **스웜/팀오케스트레이션**, 형식규율, 불변량.
- 사용자 강조 반영: F12_swarmteam(머릿속에서 병렬 sub-solver+verifier+synthesizer 시뮬).

### 인프라 미스텝 & 교훈 (중요 — 재현성)
- **`args`는 문자열로 전달된다**: Workflow의 args에 객체를 줘도 스크립트엔 JSON '문자열'로 도착.
  → 스크립트에서 `JSON.parse(args)` 필요. (첫 워크플로 `undefined` 에러의 원인)
- **권한훅 차단 = CR(`\r`)**: Python `Path.write_text`가 Windows 텍스트모드로 `\n`→`\r\n` 변환,
  `\r`(0x0d)이 승인 다이얼로그의 "hidden control character"로 차단됨. → `newline="\n"`으로 해결.
  (Write 툴 산출 스크립트는 LF만 써서 통과했던 것.) 또한 텍스트모드 read가 `\r`을 숨겨 디버깅을 지연시킴 → 바이너리로 확인.
- **노드 상한**: 워크플로당 누적 1000 에이전트. 112×10=1120 > 1000 → **5샤드 분할**.

### 병렬 팀 배치 (사용자 지시: 동시 병렬 팀으로 주간 한도 활용)
- 스크리닝 5샤드(각 ~22설계 × 어려운 10문항=P1,P4,P6,P11,P12,P14,P16,P17,P19,P20) + SEED-20(시드×20×3시행).
- = **6개 워크플로 팀 동시 가동, ~1360 평가 노드 in-flight.**
- 다음: 샤드 결과 병합 → 상위 ~16 선별 → 전체 20문항 정밀평가 → 결선(다수결) → 챔피언 통합.

### Gen2 스크리닝 결과 (어려운 10문항) — lessons & missteps
- **misstep(인프라)**: 6팀 동시(~1360노드)가 **서버 레이트리밋**을 유발 → 다수 노드 null.
  **lesson**: 동시 워크플로 수를 1~2개로 제한(워크플로당 ~14 동시). 노드 폭주 금지. → 사용자 지시 "천천히".
- **풀커버리지 만점 13종**: F02_verify_smallcase, F02_verify_dual_track, F01_decomp_b,
  F02_verify_bayes_table, F11_metacog_b, F01_decomp_c, F01_decomp_h, F02_verify_three_vote,
  F01_decomp_d, F03_adversarial_c, F01_decomp_e, F02_verify_complement, F02_verify_invert.
- **핵심 신호**: 검증(F02) 계열이 상위 다수. 점수 손실은 전부 **P4(무거운 산술)**에서 발생
  → 단일패스 분해의 산술 취약성 재확인, 검증/다수결이 방어. (KNOWHOW B항 보강)
- SEED-20(시드×20×3): baseline도 다수결 20/20. 모델이 강력 → 변별은 "견고성(robustness)"에서 갈림.
- 결정: **결선 = 상위 ~10 만점자 + baseline 레퍼런스**, 전체 20문항, **단일 워크플로**로 천천히.

## Generation 3 — 결선 (12종 × 20문항, 단일 워크플로) (2026-06-27)
- **실패 0건** (단일 워크플로 → 레이트리밋 없음). 240노드, 3.65M 토큰. → "동시 1팀" 원칙 검증됨.
- **20/20 만점 4종**: F02_verify_smallcase, F02_verify_dual_track, F03_adversarial_c, C2_selfverify.
- 19/20 (대부분 P4 콜라츠합에서 1슬립): F01_decomp_b/c/d/h, F02_verify_bayes_table(P18 편집거리 슬립),
  F11_metacog_b(P4 8417, off-by-one), F02_verify_three_vote, **C0_baseline도 19/20**.
- 결론: **검증 계열이 3세대 연속 최강.** decomp 계열은 P4에서 반복 슬립 → 산술 견고성이 핵심 변별점.

### Generation 4 — 결정전(tiebreak, 진행중)
- 4 공동챔피언 × 오류多 6문항(P1,P4,P11,P14,P18,P19) × 5시행 = 120노드. 안정성으로 단일 챔피언 선정.
- 가설: 순수 검증(F02) 계열이 P4 같은 무거운 산술에서 가장 안정적일 것.

### Gen4 결정전 결과 (4 챔피언 × 6 오류多 × 5시행 = 120노드, 0실패)
| 설계 | 단일샷 | 다수결 | 슬립 |
|------|--------|--------|------|
| **F03_adversarial_c** ★챔피언 | **29/30** | 6/6 | P4=8419(1회) |
| C2_selfverify | 29/30 | 6/6 | P4=8422(1회) |
| F02_verify_smallcase | 28/30 | 6/6 | P1,P4 |
| F02_verify_dual_track | 27/30 | 6/6 | P4,P18×2 |
- **결정적 발견**: **다수결(5시행) → 4종 전부 6/6 완벽.** 검증+self-consistency가 무거운 산술 슬립을 0으로.
- **챔피언 = F03_adversarial_c** (적대적 자기반박; 단일샷 최강 동급, GA 발굴). 배포 레시피 = **검증 + 다수결(3)**.
- 슬립은 전부 P4(콜라츠 합) — 무거운 산술이 유일 변별점. 도구사용 모드면 코드검증으로 100% 가능.
